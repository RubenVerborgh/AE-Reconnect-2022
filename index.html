Within a landscape of silos,
you have to move either data or people.

  • Big Data thinking
  • Data empowering people

Solid aims to restore choice
by separating data from apps.

  • Typical platforms nowadays store data
    inseparably from an application.
    • If we need to access the data, we cannot choose our app.
    • If we need to use the app, we can’t choose our data source.
  • By separating data from apps,
    we create independent choices.
    • In essence, we‘re taking Web apps back to desktop mode,
      where one file can be opened by multiple apps.

Every piece of data created by a person
or about them, is stored in a data pod.

Apps and services appear similarly,
but they blend data from many sources.

A person can grant apps and people access
to very specific parts of their data.

Separating app and storage competition
creates better offerings for all parties.

By abandoning data harvesting,
we restore permissionless innovation.

Solid is not a platform to replace others,
but a way of building for the Web.

  • Solid is an ecosystem.
    • Standards enable interoperability.
  • Solid is a movement.
    • We need to shift the app builder mindset.
  • Solid is a community.
    • Building Solid requires different people,
      companies, and organisations.

Solid enables interoperable data vaults
through standardization.

  • Solid builds on existing Web standards.
  • The standards cover:
    • data pods
    • applications or services
    • identity (authentication and authorization)
  • Solid specifies the interactions
    between data, apps, and identities.

Client–server communication is governed
by the Solid specifications.

  • The specifications are a set
    of multiple documents.
  • They essentially demand compliance
    with a couple of existing standards:
    • HyperText Transfer Protocol (HTTP)
    • Linked Data Platform (LDP)
    • Linked Data Notifications (LDN)

Several open-source implementations
of servers, apps, and libraries exist.

  • server
    • Use a public instance or host your own.
  • apps
    • data browser
    • contacts
    • photos
    • meeting organizer
    • …
  • libraries
    • authentication
    • data processing
    • …

A Solid server acts as a data pod
that stores and guards your data.

  • It is a regular Web server
    • …with support for access control.
    • …with support for Linked Data.
  • Its interface is application-agnostic.
    • Build any application, like on the Web.
    • Application-specific logic resides in clients.
  • A Solid pod is essentially a website with data.
    • Its data is compatible with any Solid app.

Solid servers separate apps from data
by implementing a set of specifications.

  • Think of a data pod as a USB stick.
    • connector shape
    • electrical signals
    • file system format
    • …
  • The Solid specifications ensure that
    data can be accessed by any Solid app
    regardless of the server that stores it.

All Solid servers must implement the spec,
yet that doesn’t make them all the same.

  • All USB sticks behave the same, but variations exist.
    • different storage capacity
    • different sizes and colors
    • different security mechanisms
    • …
  • Similarly, there are many ways to implement
    a compatible server of Solid data pods.
    • Use cases inform the need for pods.

The Community Solid Server should
grow along with developers’ needs.

  • 100% compatible with the specifications
    • anticipate envisaged spec evolutions
  • helpful companion for app developers
    • lightweight and easy to configure
  • support extension and experimentation
    • testing bed for the future of Solid

The Community Server is implemented
in TypeScript for Node.js.

  • App developers have a JavaScript-oriented setup,
    so running the server will feel familiar.
  • Common code can be run on either
    or both the client or the server.
  • Typings foster collaboration
    by making contracts explicit.

The architecture is designed
to realize a modular design.

  • It focuses on the core task of a Solid server:
    authenticated Linked Data Platform requests.
  • We created an HTTP request flow with
    processing steps and data elements.
    • This inspired the object-oriented design.
  • We emphasize the single-responsibility principle.
    • Plugging components in and out becomes routine.
